一、数据类型

由于Java是一种强类型语言，意味着必须为每一个变量声明一种类型。在Java中一共有8种基本类型，其中有4种整型、2种浮点型、1种用于表示Unicode编码的字符的单元的字符类型char、1种用于表示真值的boolean。

1. 整型

整型用于表示没有效数部分的数值，允许为负数。Java提供了4种整型，具体如下表

类型 存储需求 取值范围
int 4字节 -2 147 483 648~2 147 483 647
short 2字节 -32 768~32 767
long 8字节 -(2的32次方)~(2的32次方)
byte 1字节 -128~127

长整型(long)数值有一个后缀L(如：200L)。十六进制数值有一个前缀0x(如：0xCAFE)。八进制有一个前缀0(如：010 -> 十进制的8)，八进制容易混淆，建议不要使用八进制常数。Java 7开始，加上前缀0b可以写二进制数(如：0b1001 -> 9)，另外同样从Java 7开始，还可以为数字字面量加下划线，如1_000_000表示一百万，Java编译器会自动去除这些下划线。

2. 浮点型

浮点类型用于表示有小数部分的数值。Java有2种浮点类型，具体如下

类型 存储需求 取值范围
float 4字节 大约 ± 3.402 823 47E + 38F (有效位数6~7位)
double 8字节 大约± 1.797 693 134 862 315 70E + 308 (有效位数15位)

double表示这种类型的数值精度是float类型的两倍(有人称为双精度数值)。绝大部分情况下用double类型。float类型的数值有一个后缀 F (例如，3.14F)。没有 F 的浮点数值默认为double类型。

3. char类型

char类型用于表示单个字符。通常用来表示字符常量。例如：’A’是编码为65所对应的字符常量。与”A”不同，”A”是一个包含字符的字符串。Unicode编码单元可以表示为十六进制值，范围从\u0000到\Uffff。例如：\u03C0表示派。

类型 存储需求 取值范围
char 2字节

4. boolean类型

boolean类型有两个值：false和true，用来判定逻辑条件。在编译时会被编译成int类型，使用 1 来表示 true，0 表示 false。

类型 存储需求 取值范围
boolean 1字节

二、字符串

Java中操作字符串可以使用String、StringBuilder、StringBuffer，这三个类

1. String

看一下Java中String类是什么样子的~

public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    /** The value is used for character storage. */
    private final char value[];
    ......
    // 看一下String中的equals()
    public boolean equals(Object anObject) {
    	// 先判断两者是否为同一个对象的引用，如果是直接返回true
        if (this == anObject) {
            return true;
        }
        // 判断传入的对象是否为String类型，如果不是直接返回false
        if (anObject instanceof String) {
            String anotherString = (String)anObject;
            int n = value.length;
            // 这里n是当前String对象的value数组的长度，如果两者长度不同直接返回false
            if (n == anotherString.value.length) {
                char v1[] = value;
                char v2[] = anotherString.value;
                int i = 0;
                // 长度相同，那么就逐位比较是否为同一个字符，如果有任意一个字符不同则直接返回false
                while (n-- != 0) {
                    if (v1[i] != v2[i])
                        return false;
                    i++;
                }
                // 每一位字符都相同则返回true
                return true;
            }
        }
        return false;
    }
}

可以看到String被final修饰，所以String是不能被继承的。同时value数组也被final修饰，确保了String是不可变的。在Java 8中String内部使用char类型的数组来存储数据。

在Java 9中改为使用byte类型的数组来存储数据

字符串常量池 (String Pool )

说到String就不得不说一下字符串常量池了。

字符串常量池保存着所有字符串字面量，这些字面量在编译时期就确定。而且，还可以使用String的 intern() 方法在运行过程动态的将String添加到 String Pool 中。

intern()

当一个字符串调用 intern() 方法时，如果 String Pool 中已经存在一个字符串和该字符串值相等，那么就会返回 String Pool 中字符串的引用；否则，就会在 String Pool 中添加一个新的字符串，并返回这个新字符串的引用。

举个🌰

String s1 = new String("szw");
String s2 = new String("szw");
System.out.println(s1 == s2);           // false
String s3 = s1.intern();
String s4 = s2.intern();
System.out.println(s3 == s4);           // true
System.out.println(s1 == s3);           // false

上面这个🌰中，首先s1和s2是通过new的方式创建字符串，故而会直接在堆内存中分配，所以s1 == s2是false。

String str = new String(“xxx”)

使用上面这个方法创建字符串时，会创建两个字符串对象。一个是通过new在堆内存上分配的对象，另一个就是因为”xxx”属于字符串，而在字符串常量池中创建的对象(前提是字符串常量池中没有”xxx”这个对象)

其次，无论调用s1.intern()还是s2.intern()都发现字符串常量池中有”szw”直接返回这个引用，所以s3 == s4是true。

最后，因为s1的引用是在堆内存中，而s3的引用是在字符串常量池中，所以s1 == s3是false。

在 Java 7之前，String Pool 被放在运行时常量池中，属于永久代。而从 Java 7开始，String Pool 被移到堆中。这是因为永久代的空间有限，在大量使用字符串的场景下会导致 内存溢出(OutOfMemoryError)。

提个问题~

String s = "szw";
s = s + "799";

请问上面的原始的String对象是否发生了改变？

答案：没有，准确来说是”szw”这个对象没有发生改变，改变的是s指向了一个新的String对象，内容是”szw799”。

因为String是不可变的呀，value数组可是被final修饰的啊。所以，在进行 + 拼接之后s就会指向一个新的字符串~

2. StringBuilder和StringBuffer

首先，StringBuilder和StringBuffer的共同点是：它俩都是可变的字符串对象。

其次，StringBuilder和StringBuffer的不同点是：前者是线程不安全的，而后者是线程安全的(使用synchronized保证线程安全)。

StringBuilder是从JDK 1.5开始引入的，所以在JDK 1.5之间不要使用StringBuilder。

话说，还有人使用JDK 1.5吗。。。

+ 和 StringBuilder/StringBuffer 中的append()方法，都可以拼接字符串，使用那个好呢？

看下面这段代码

String s = "abc";
String ss = "ok" + s + "xyz" + 5;
System.out.println(ss);

通过反编译后看下面这段代码

//    0    0:ldc1            #16  <String "abc"> 
//    1    2:astore_1         
String ss = (new StringBuilder("ok")).append(s).append("xyz").append(5).toString(); 
//    2    3:new             #18  <Class StringBuilder> 
//    3    6:dup             
//    4    7:ldc1            #20  <String "ok"> 
//    5    9:invokespecial   #22  <Method void StringBuilder(String)> 
//    6   12:aload_1
//    7   13:invokevirtual   #25  <Method StringBuilder StringBuilder.append(String)> 
//    8   16:ldc1            #29  <String "xyz"> 
//    9   18:invokevirtual   #25  <Method StringBuilder StringBuilder.append(String)> 
//   10   21:iconst_5 
//   11   22:invokevirtual   #31  <Method StringBuilder StringBuilder.append(int)> 
//   12   25:invokevirtual   #34  <Method String StringBuilder.toString()> 
//   13   28:astore_2   
// 后面就不贴了~

通过上面我们发现虽然在源程序中使用了”+”，但在编译时仍然将”+”转换成StringBuilder。因此，我们可以得出结论，在 Java 中无论使用何种方式进行字符串连接，实际上都使用的是 StringBuilder。

如果连接字符串行表达式很简单（如上面的顺序结构），那么”+”和StringBuilder基本是一样的，但如果结构比较复杂，如使用循环来连接字符串，那么产生的 Java Byte Code 就会有很大的区别。

看下面这段代码

String s = ""; 
Random rand = new Random();
for (int i = 0; i < 10; i++) { 
	s = s + rand.nextInt(1000) + " ";
}
System.out.println(s);

反编译后的代码我就不贴了~直接说结论吧，上面这段代码会在for循环内部创建10个StringBuilder对象，极大的消耗了性能。

如果改成下面这种

StringBuilder result = new StringBuilder(); 
Random rand = new Random();
for (int i = 0; i < 10; i++) { 
	result.append(rand.nextInt(1000));
    result.append(" ");
}
System.out.println(result.toString());

由于只有一个StringBuilder，所以会有更高的效率，同时消耗的资源也更少了~

如果将上面的代码在JDK1.4下编译，必须将StringBuilder改为StringBuffer，而JDK1.4将”+”转换StringBuffer。StringBuffer和StringBuilder的功能基本一样的~
